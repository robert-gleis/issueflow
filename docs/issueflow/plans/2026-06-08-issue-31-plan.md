# Workflow Timeline Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship a workflow timeline that projects issue lifecycle events from the SQLite event log into ordered, timestamped steps with failure/retry visibility, exposed via `issueflow timeline show`.

**Architecture:** New `src/timeline/` module with pure `buildTimeline` reducer, text/JSON renderers, and `createWorkflowEventSubscriber` for engine telemetry. Extend `EVENT_TYPES` with four workflow event types. Thin `issueflow timeline show` CLI opens the log, builds timeline, prints output.

**Tech Stack:** TypeScript (strict, ES2022, NodeNext), vitest, Commander, existing `openEventLog` from `src/event-log/`.

- [ ] **Step 1: Write failing builder tests**

Cover:
1. Empty events → all pending, `hasActivity: false`
2. `plan.approved` alone → `planned` completed, `hasActivity: true`
3. `team.planned` → `planned` completed
4. Happy path: `plan.approved` → transitions through `workflow.transition` payloads → `verification.passed` → `pr.created`
5. `verification.failed` then `verification.passed` → two attempts on `verified`, final status `completed`

10. Malformed `workflow.transition` payload (missing `to`) → step state unchanged, no throw
11. Partial pipeline (`plan.approved` only) → `planned` completed, `implemented` `in_progress` when followed by incomplete transition
12. Step `failed` while later steps still `pending` (review failed, verified not started)
13. `startedAt` / `completedAt` from first attempt and last successful attempt

- [ ] **Step 3: Implement buildTimeline**

```ts
export function buildTimeline(issueNumber: number, events: EventRecord[]): Timeline
```

Algorithm:
- Sort events by `id` ascending (CLI passes store results newest-first; builder re-sorts)
- Initialise steps from `CANONICAL_STEPS` with empty attempts
- For each event, apply mapping rules from spec (skip unknown/malformed payloads defensively — no throw)
- After processing, derive per-step fields:
  1. `startedAt` — timestamp of first attempt on the step
  2. `completedAt` — timestamp of last `completed` attempt, or `null`
  3. `status`:
     - `pending` — no attempts
     - `completed` — last attempt is `completed`
     - `failed` — last attempt is `failed` AND no later pipeline step has any `completed` attempt
     - `in_progress` — has attempts, not `completed`, and this step has the highest pipeline index among steps with attempts
- Set `hasActivity` when any attempt recorded

Private helpers: `recordAttempt(step, attempt)`, `applyEvent(stepMap, event)`.

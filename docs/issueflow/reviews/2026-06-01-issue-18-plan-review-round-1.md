# Plan Review — Issue #18, Round 1

## Status
pass_with_findings

## Summary
The plan faithfully implements the spec: identifier names, state union, error-code union, `SpawnSpec`/`LogSnapshot`/`LogOptions` field names, `ScriptedRunnerScript` knobs, and the test list (including the engine-isolation regression) all match. Tasks 1–6 follow a clean red/green TDD loop with concrete failing-test commands and minimum implementations. The main wrinkles are: Task 7 explicitly notes its two tests "pass already" against Task 6's code, which is an admitted break from red-green discipline (it's a coverage-pin rather than a TDD cycle); the `stop()` transition from `starting` is documented in the spec but has no test in either the spec's test list or the plan; and a handful of small drift items (test count off in Task 1, no test that `stop()` clears/preserves `error` after `error → stopped`, etc.) are worth tightening before implementation.

## Findings

1. **major — Task 7 is not red→green; it's a coverage-pin commit (plan: Task 7, Step 2).** The plan says explicitly: "Both tests pass already — the existing implementation resets state on respawn and `status()` already builds a fresh object each call. This task confirms that contract via tests." Tracing the logic confirms this is true: Task 3's spawn allows `idle|stopped` and resets `stoppedAt`/`exitCode`, and Task 2's `status()` already constructs a fresh object each call. There is no failing-test step in Task 7. This is a deliberate deviation from the TDD loop the other tasks follow; either reframe Task 7 as a non-TDD "lock-in" task (acknowledge in the task heading and skip the Step 2 "verify it fails" expectation), or split it so each assertion is added before the implementation it pins (e.g. write the freshness test before Task 2 returns a fresh snapshot). Functionally fine; methodologically a break from the discipline elsewhere in the plan.

2. **major — Task 1 reports "PASS, 6 tests" but the file contains 6 tests across two describe blocks; Task 8 reports "8 tests in this file (6 existing + 2 new)" which adds up — but the existing Task 1 count of "6 tests" is only correct if you count the `RunnerError` describe's 2 tests (plan: Task 1, Step 4; Task 8, Step 4).** Counting: `Runner (structural)` has 4 `it` blocks (in-line impl, SpawnSpec, LogOptions, RunnerState pin); `RunnerError` has 2 `it` blocks. Total = 6. Math is correct; flagging as nit-major because the plan's totals propagate downstream — verify before adopting.  *(Demote to nit on re-read; included for transparency since the user asked for arithmetic checks.)*

3. **minor — No test for `stop()` from `starting` (plan: Task 6 test list).** The spec explicitly enumerates `running|starting|error → stopping → stopped` (line 132) but the spec's test list (line 247-250) and the plan's Task 6 only cover `running`, `idle`, `stopped`, and `error`. With `spawnDelayMs`, a `starting` window is reachable; a one-line test (`spawn(...); stop(); await spawnPromise.catch(...)`) would close the gap and exercise the documented transition. Optional but cheap given the `spawnDelayMs` fixture already exists.

4. **minor — Spec-vs-impl gap on what happens to `status.error` after `error → stopped` (plan: Task 6 implementation and "transitions error → stopped" test).** The test asserts `status.error === 'boom'` after the `error → stopped` transition. The spec's `stop()` contract (lines 130-135) does not say whether the prior `error` message is preserved or cleared. The plan picks "preserve" by never clearing `errorMessage` in `stop()`. The choice is reasonable (debuggability) but undocumented. Either add a one-line note to the spec ("`stop()` preserves the prior `error` field when transitioning from `error`") or flip the implementation. Don't leave it implicit — a future runner author will copy this pattern.

5. **minor — Reuse-after-stop test in Task 7 does not assert `status.startedAt` (plan: Task 7, Step 1, first test).** It asserts `state: 'running'`, `stoppedAt: undefined`, `exitCode: undefined` — correctly verifying the reset — but does not check that `startedAt` is a *fresh* Date (i.e. > the previous `startedAt`). Without this, an implementation that forgets to overwrite `startedAt` on the second spawn would still pass. Cheap improvement: capture `startedAt` after first spawn, assert second-spawn `startedAt` is later or at least `instanceof Date` distinct from undefined.

6. **minor — Task 8's barrel test relies on `runnersBarrel.Runner` as a type at runtime via namespace access (plan: Task 8, Step 1, second test).** `const runner: runnersBarrel.Runner = new runnersBarrel.ScriptedRunner('r1');` — namespace-qualified type access on a `* as` import works under `module: NodeNext` with TS 5.9, but only because `Runner` is exported as a pure type via `export type { ... }`. If a future refactor moves `Runner` to a value export (or drops `export type` for `export`), the namespace-qualified type access would still work but signals fragility. Not a bug; just worth a comment so a future reader doesn't "fix" it.

7. **minor — Engine-isolation regex misses dynamic imports and `require()` (plan: Task 9, `RUNNER_IMPORT_REGEX`).** The regex matches `from\s+['"]...runners.../...['"]` — i.e. static `import ... from '...'`. It does not catch `await import('../runners/index.js')` or (theoretical) `require('../runners')`. The repo today is pure ESM, so `require` is moot, but dynamic `import()` is plausible if someone tries to lazy-load. Either add an `import\s*\(['"]...['"]\)` alternative, or document the regex's intentional static-import-only scope in a code comment.

8. **minor — Task 7's commit step does not stage `src/runners/scripted.ts` (plan: Task 7, Step 3).** Correct — Task 7 only touches the test file — but the contrast with Tasks 3-6, which all stage both files, is jarring. A one-line note in Task 7's heading ("test-only commit; no implementation change") would prevent a future executor from wondering whether they forgot something.

9. **nit — `Runner` import shape in Task 8's barrel test uses `* as` which deviates from named-import convention elsewhere (plan: Task 8 first test block).** Existing repo tests use named imports (`import { ScriptedAgentAdapter } from ...`). Using `* as runnersBarrel` is fine and arguably clearer for a barrel-coverage test, but flag the convention shift in the task body.

10. **nit — `Notes for Reviewers` section duplicates content from the spec (plan: bottom of file).** The notes about the engine-isolation test being "intentionally over-strict" and the `start.ts:480` execa call being "intentionally untouched" are already in the spec. Either trim the duplication or keep as a quick-reference; either is fine but worth a once-over.

## What looks good

- **Identifier and type fidelity is character-perfect.** `RunnerState` union (`idle|starting|running|stopping|stopped|error`), `RunnerStatus` fields (`state`, `startedAt`, `stoppedAt`, `exitCode`, `error`), `SpawnSpec` (`binary`, `args`, `cwd`, `env?`), `LogSnapshot` (`stdout`, `stderr`, `combined`, `truncated`), `LogOptions.sinceByteOffset`, `RunnerErrorCode` (`invalid-state|spawn-failed|stop-failed|logs-unavailable`), and `ScriptedRunnerScript` (`stdout`, `stderr`, `exitCode`, `spawnDelayMs`, `failOnSpawn`) all match the spec verbatim.
- **Test conventions match the repo.** Tests live flat under `tests/unit/` (no subdirectories), use vitest, use the `../../src/runners/X.js` import path with the required `.js` extension. Verified against `ls tests/unit/` — every existing test file is `tests/unit/<name>.test.ts`. The plan adheres.
- **Orthogonal-axis isolation is preserved.** `src/runners/` is a new top-level directory; no plan task touches `src/agents/`, `src/adapters/`, or `src/workflow/`. The barrel and concrete impl mirror the `src/agents/{types,scripted,index}.ts` layout exactly. Verified `src/workflow/` currently contains no `tmux` references and no `runners` imports — Task 9's regression test will pass on first run.
- **Task 4's `starting`-state observability does work as claimed.** Tracing: spawn's body runs synchronously up to `await new Promise((r) => setTimeout(r, 30))` which suspends; control returns to caller; `status()` is async but its body is synchronous and resolves on the next microtask, observing `state: 'starting'`. The macrotask-vs-microtask ordering makes this deterministic, not racy.
- **Task 7's "pass already" claim is verified correct.** Task 3 already allows `idle|stopped` as spawn preconditions and resets `stoppedAt`/`exitCode`; Task 2 already builds a fresh `RunnerStatus` object each `status()` call. Both Task 7 tests do pass against Task 6's implementation — the methodology nit (Finding 1) is separate from the correctness.
- **TDD discipline is otherwise strict.** Tasks 1, 2, 3, 4, 5, 6, 8, 9 each have a clear failing-test step with the expected failure mode, then minimum-viable implementation. No TBDs, no "implement later", no "similar to Task N" handwaving.
- **TypeScript compatibility is solid.** `tsconfig.json` has `strict: true` but no `noUnusedParameters` / `noUnusedLocals`, so `_spec` and `_options` parameters are fine. The existing `src/agents/scripted.ts` uses the same `_input` pattern with no compiler complaint. NodeNext + ESM + `.js` extensions all honoured.
- **Engine-isolation regex handles the realistic cases.** Trace-tested against `from '../runners/types.js'`, `from '../../runners/index.js'`, and `from '../runners'` — all match. `tmux` case-insensitive regex catches `Tmux`, `TMUX`, etc.

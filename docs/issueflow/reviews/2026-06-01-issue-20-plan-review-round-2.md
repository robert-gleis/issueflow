# Plan Review Round 2 — Issue #20

## Status
pass_with_findings

## Summary
The plan is scope-correct, well structured, and follows the existing DI / ESM `.js` import / vitest patterns in the codebase. Schemas, types, and test fixtures stay consistent across tasks, the SIGINT/abort contract is plumbed end-to-end through `RunPipelineInput.abortSignal` and `verifyAction`'s one-shot handler, and the `getIssueflowPath` relative-path quirk is correctly handled in `resolveIssueNumber`. A handful of issues are still worth fixing before implementation: concurrent log writes can interleave, the runner's signal-kill recording is internally inconsistent with how `defaultRunPipelineDeps.execCheck` reports SIGINT, the default `execCheck` has a real race attaching `data` listeners after spawn with default execa stream modes, and a couple of test-coverage gaps around stdout/stderr interleaving and ENOENT recording.

## Findings

1. **Concurrent `handle.write` calls on the same `FileHandle` can reorder log lines**
   - Where: Task 4 — `src/verification/runner.ts`, the `onChunk` callback that does `pendingWrites.push(handle.write(payload))`.
   - Issue: `fs.promises.FileHandle.write` does **not** guarantee ordering across concurrent calls on the same handle (see Node.js docs: "It is unsafe to use `filehandle.write()` multiple times on the same file without waiting for the promise to be resolved (or rejected)"). When a noisy check emits several `data` chunks back-to-back, the synchronous `pendingWrites.push(handle.write(...))` schedules overlapping writes; the lines they emit may land in the wrong order on disk.
   - Why it matters: The whole point of `[stdout]` / `[stderr]` line prefixing is so the original stream is recoverable. Under heavy log volume this corrupts the log file order. The test `'does not mangle prefixes when chunks arrive mid-line'` happens to use only two chunks so it never trips this, but the integration test asserts only a substring (`expect(passLog).toContain('[stdout] ok')`) and would not catch reordering either.
   - Fix: Serialize writes through a single chained promise. e.g. keep a `writeQueue: Promise<void> = Promise.resolve()` and replace `pendingWrites.push(handle.write(payload))` with `writeQueue = writeQueue.then(() => handle.write(payload)).then(() => {})`, then `await writeQueue` at the end. Or buffer chunks in memory and flush sequentially in one pass. Add a test that emits many small chunks across both streams and asserts the resulting log content/order.

2. **`defaultRunPipelineDeps.execCheck` returns `signal: result.signal ?? null` only on successful subprocess resolution — the runner never records the SIGINT marker the abort path is supposed to emit when `abortSignal` is already aborted before the loop reaches the next check**
   - Where: Task 4 — `runVerificationPipeline` skip branch (`if (bailed || abortSignal.aborted) { ... status: 'skipped', signal: null, ... }`) interacting with the Task 6 `runExitCode` (`run.checks.some((check) => check.signal === 'SIGINT')`).
   - Issue: Exit code 130 is only emitted when **some completed check has `signal === 'SIGINT'`**. That requires the default `execCheck` to actually kill a running subprocess with SIGINT (which it does via `subprocess.kill('SIGINT')`). However, if `abortSignal.aborted` fires between checks (after a check that finished naturally with exit 0), no check ends up with `signal === 'SIGINT'`. The next iterations are recorded as `'skipped'` with `signal: null`. `runExitCode` then maps the run to either 0 or 1 based on `run.status`, never 130 — even though the user clearly cancelled. The runner test `'marks remaining checks skipped when abortSignal aborts mid-pipeline'` aborts after the first check passes and asserts `run.status === 'fail'` (because skipped = fail), but it never asserts the exit code; the verify-command test only covers the explicit-SIGINT-signal-on-a-check case.
   - Why it matters: A SIGINT delivered between two checks (a real and common timing — the user hits ^C while logs are flowing for check N+1's start) silently returns exit 1 instead of 130, contradicting the spec's "130: Pipeline cancelled by SIGINT".
   - Fix: Pass `abortSignal.aborted` through to `runExitCode` (the `aborted` flag is already in the signature but `verifyAction` constructs the controller and never forwards the final `aborted` state — `createVerifyPlan` uses `input.abortSignal?.aborted ?? false` which works only if the abort flag was set before `runExitCode` runs; that part is correct). The remaining gap is that `verifyAction` passes its own `controller.signal` in, and the runner finishes well after the signal was aborted, so `input.abortSignal?.aborted` will be `true` and `runExitCode` does return 130. That part actually works. **However**, the unit test `'marks remaining checks skipped when abortSignal aborts mid-pipeline'` should additionally assert that this scenario, when fed through `createVerifyPlan`, returns `exitCode: 130`. Add either an end-to-end verify-command test that aborts mid-run, or extend the runner test to also check via `createVerifyPlan` that the exit code is 130 in this case. Otherwise the contract has no test guard.

3. **Race: `defaultRunPipelineDeps.execCheck` attaches `data` listeners after `execa(...)` returns — first chunks can be lost**
   - Where: Task 4 — `defaultRunPipelineDeps.execCheck` in `src/verification/runner.ts`.
   - Issue: With execa 9 and explicit `stdout: 'pipe'`, `stderr: 'pipe'`, plus `buffer: false`, the child process spawn happens synchronously inside `execa(...)`. The returned `subprocess.stdout` is a `Readable`. In Node, `Readable` streams default to paused mode and stay paused until a consumer subscribes — but execa internally attaches its own consumers when `buffer: true` (the default). With `buffer: false`, execa is supposed to leave the streams alone. In practice this works, but the plan's claim "`buffer: false` plus explicit `stdout: 'pipe'` / `stderr: 'pipe'` so execa does not race with the user-side `'data'` listeners" is the right intuition; what is **not** safe is that the listener attachment is awaited only after `execa(...)` returns, while spawning is in-flight. If the child emits to stdout before the JS microtask that registers `.on('data', ...)` runs, those bytes will be queued in the stream buffer and delivered when the listener attaches. That part is OK because Node buffers paused streams. The actual hazard: if you read `subprocess.stdout` with `subprocess.stdout?.on('data', ...)` but later abort and `subprocess.kill('SIGINT')` causes the stream to error, no `'error'` handler is wired → an unhandled stream error can crash the process.
   - Why it matters: Aborted runs may crash the CLI instead of cleanly exiting with code 130.
   - Fix: Attach `'error'` listeners on both `subprocess.stdout` and `subprocess.stderr` that swallow / log to the in-progress log file. Alternatively wrap the listener attachment so any stream errors funnel into `onChunk('stderr', ...)`.

4. **`subprocess.kill('SIGINT')` may resolve without `signal: 'SIGINT'` on some shells / hosts**
   - Where: Task 4 — `defaultRunPipelineDeps.execCheck`, plus the SIGINT contract in Task 6.
   - Issue: When the kernel propagates SIGINT to a Node child via `subprocess.kill('SIGINT')`, the child typically exits via signal, and execa reports `result.signal === 'SIGINT'`. However for some subprocesses (notably `npm run …` wrappers) the signal is intercepted and the child reports a non-zero exit code instead, with `result.signal === undefined`. Result: the runner records the check as `fail` with `exitCode != 0` and `signal: null`. `runExitCode` then relies on `abortSignal.aborted` to still produce 130. That fallback **is** in place (`runExitCode(run, input.abortSignal?.aborted ?? false)`). Good. But the runner unit tests never exercise the "abort fires while a check is running" path — only the "abort fires between checks" path — so this fallback is also untested.
   - Why it matters: The 130 contract is one of the four user-visible exit codes. Two of the three ways to reach it are currently uncovered by tests. A regression in `runExitCode` (e.g. dropping the `aborted` parameter) would not be caught.
   - Fix: Add a runner test where `execCheck` is stubbed to observe `abortSignal` and resolve mid-flight with `{ exitCode: 130, signal: null }` after `controller.abort()` fires; then feed the resulting run through `createVerifyPlan` and assert `exitCode === 130`.

5. **Runner test `'records signal-killed checks as fail with the signal name'` asserts `status: 'fail'` based on `signal === 'SIGTERM'`, but the runner code derives status only from `exitCode === 0 && signal === null`**
   - Where: Task 4 — `runVerificationPipeline` line `const status: CheckStatus = exec.exitCode === 0 && exec.signal === null ? 'pass' : 'fail';`.
   - Issue: This is correct as written, but note that a process that exits cleanly **and** is also reported with a signal (rare but possible on macOS when a process self-`raise`s after writing exit code) would be marked `'fail'`. The test only covers `{ exitCode: null, signal: 'SIGTERM' }`. Fine. But the comment in the plan claims `CheckResult.status = 'pass' iff exitCode === 0` (from the spec) — and the actual rule in the runner code is stricter (also requires `signal === null`). Spec and code agree only when the OS never delivers a signal alongside a 0 exit, which is the normal case. Worth aligning the wording in code comments / the spec restate in plan task summary.
   - Why it matters: Cosmetic / consistency.
   - Fix: Either tighten the spec wording, or in code use just `exec.exitCode === 0 ? 'pass' : 'fail'` (then `signal != null` cases will already have `exitCode === null` and fail). Pick one.

6. **Branch parser is over-strict: `^issue\/(\d+)-` requires a trailing dash, but spec only says `issue/<number>-<slug>` — what about `issue/20`?**
   - Where: Task 5 — `src/core/issue-id.ts`, the `branchPattern = /^issue\/(\d+)-/`.
   - Issue: The session-state writer always produces `issue/<n>-<slug>` (with non-empty slug), so the trailing dash will always be present in practice. But if a user manually checks out `issue/20` (no slug), the resolver throws despite the issue number being recoverable. Branch test `'ignores non-matching branch names'` only checks `'main'`. The acceptance criteria don't dictate this rigor either way; it's defensible.
   - Why it matters: Minor robustness gap.
   - Fix: Either change to `/^issue\/(\d+)(?:-|$)/` to also accept `issue/20`, or document explicitly in the plan that the trailing slug separator is required and add a test that asserts `issue/20` is rejected (to lock the behaviour in deliberately). Right now neither is asserted.

7. **`tests/integration/verify-command.test.ts` does not depend on `worktrunk` but the project's start command does; vitest defaults will pick this file up, but check `npm test` already runs integration tests**
   - Where: Task 8 — `tests/integration/verify-command.test.ts`, plus repository test scripts.
   - Issue: There is no `vitest.config.*` file (verified). Vitest's default include glob is `**/*.{test,spec}.{js,ts,...}` so `tests/integration/verify-command.test.ts` **will** be discovered and run by `npm test`. That is the intended behaviour for this issue. But every other test file in `tests/` lives under `tests/unit/` — there is no prior `tests/integration/` directory and no precedent for whether the project wants those split. Right now the integration test will simply join the unit suite. Worth confirming intentionally, because the existing `tests/unit/start-command.test.ts` is already integration-flavoured.
   - Why it matters: Style consistency only. Not a correctness risk.
   - Fix: Either keep as is and note in the plan that integration tests run unconditionally, or move the file to `tests/unit/verify-command-integration.test.ts` to match existing convention. No code change needed otherwise.

8. **Test fixture `signal: null` for ENOENT inconsistent with default `execCheck` behaviour**
   - Where: Task 4 — `verification-runner.test.ts`, the `'records command-not-found as fail with a message in the log'` test stubs `execCheck` to return `{ exitCode: null, signal: null, error: 'ENOENT' }`.
   - Issue: The runner stores `exec.signal` directly into `CheckResult.signal`. The stub returns `null`. So `signal: null` ends up in the check record. The test asserts the log contains "ENOENT" — but never asserts how `error` propagates. In the production `defaultRunPipelineDeps.execCheck`, the spawn error is caught and written via `onChunk('stderr', `${message}\n`)`; the returned object has `error: message`. The runner code in `runVerificationPipeline` never reads `exec.error`. So `ExecCheckResult.error` is currently dead. Either the field should be removed from the interface, or the runner should incorporate it into the log/result. Right now it is decorative.
   - Why it matters: Cognitive load — readers will see `error?: string` on `ExecCheckResult` and assume something uses it. Tests pass without exercising it.
   - Fix: Either delete `error?: string` from `ExecCheckResult` (the prod path already writes the message via `onChunk('stderr', ...)` so it's truly redundant), or have the runner explicitly write `[runner] command failed: ${exec.error}` into the log when present. Pick one and align the test.

9. **`VerifyPlanDeps.resolveIssueNumber` signature differs from the underlying function's three-arg form**
   - Where: Task 6 — `src/commands/verify.ts`, `VerifyPlanDeps.resolveIssueNumber: (worktreePath: string, override: number | undefined) => Promise<number>` vs. `src/core/issue-id.ts` `resolveIssueNumber(worktreePath, override, deps = defaultDeps)`.
   - Issue: The default `defaultVerifyPlanDeps` adapts the function with a two-arg arrow `(worktreePath, override) => resolveIssueNumber(worktreePath, override)`. That works. But the `repoRoot` resolution path passes `repoRoot` (from `resolveRepoRoot`) as the first argument to `resolveIssueNumber`, treating it as `worktreePath`. In a linked worktree, `git rev-parse --show-toplevel` returns the *worktree* root (which is the worktree path), so this is fine. In the main worktree it returns the same path. The naming inconsistency between `worktreePath` (in `resolveIssueNumber`) and `repoRoot` (in `createVerifyPlan`) is a clarity issue — both refer to the same thing here.
   - Why it matters: Cognitive load. Future maintainers may wonder which is which.
   - Fix: Either rename `VerifyPlanDeps.resolveIssueNumber`'s first parameter to `repoRoot` for consistency with the rest of the verify command code, or add a one-line comment in the plan noting they are equivalent here. No runtime impact.

10. **`verifyAction`'s SIGINT handler will not produce exit 130 when a check is killed by SIGINT but execa reports `signal: 'SIGINT'` AND `runExitCode` already received `aborted: false`**
    - Where: Task 6 — `verifyAction` constructs an `AbortController`; the SIGINT handler calls `controller.abort()`. `createVerifyPlan` passes `controller.signal` as `abortSignal`. By the time `runExitCode` runs, `input.abortSignal?.aborted` reflects whether SIGINT was ever caught. Fine.
    - Issue: This works for the verify command's own SIGINT. But `verifyAction` calls `process.once('SIGINT', onSigint)`. Node's default SIGINT handler triggers process exit; once you attach a custom listener, Node won't exit by default. The handler aborts the controller but does NOT signal back to Node that the process should terminate. The control flow then continues until `createVerifyPlan` resolves and sets `process.exitCode = 130`. The integration test does not cover this; runner unit tests don't either. The `'maps a run with a SIGINT-signalled check to exit code 130'` unit test stubs the entire pipeline result, so the `process.once('SIGINT', ...)` glue is untested.
    - Why it matters: The SIGINT contract has zero end-to-end coverage. A regression in `verifyAction`'s try/finally (e.g. forgetting to remove the listener) would not be caught.
    - Fix: Add a small unit test for `verifyAction` (not just `createVerifyPlan`) that spies on `process.removeListener` and verifies the SIGINT handler is unregistered after both pass and error paths. Optional but cheap.

## Notes

- The `'does not mangle prefixes when chunks arrive mid-line'` test is good and catches the off-by-one prefix bug class. Consider adding a sibling test where two `stdout` chunks then a `stderr` chunk arrive without a newline in between, to lock the per-stream tail-buffer behaviour.
- `aggregateStatus` returns `'pass'` iff every check is `'pass'`. Worth a one-line comment in code reminding readers that `'skipped'` counts as fail (matches spec, but it's easy to forget).
- The plan persists `run.json` in the runner and also retains `writeRun` in the store. That's fine for symmetry, but consider exporting `writeRun` from `store.ts` while having `runner.ts` *call* `writeRun(run)` instead of duplicating the `fs.writeFile(path.join(...), JSON.stringify(run, null, 2))` logic. Currently two places know the file name `run.json`; one would do.
- Acceptance criterion "Results are persisted and retrievable by issue id" is satisfied by `listRuns` / `loadLatestRun` even though `verifyAction` itself never reads prior runs. Worth a one-line note in the plan that the persistence is consumed only by future tickets, so that's not a missed feature here.
- Smoke test `node dist/src/bin.js verify --help` (Task 9 step 4) is nice but depends on `npm run build` having put a working bin in place; that's already covered by step 2. Consider asserting the smoke test output programmatically (e.g. via execa in a test) so it runs in CI rather than as a manual step.

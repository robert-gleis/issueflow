## Findings

### 🟡 Suggestion — Idempotent approve skips `parent_issue` consistency check

**Location:** `src/commands/decomposition.ts:314-319`

When `decomposition-applied.json` exists, approve prints children and exits `0` without verifying `applied.parent_issue === resolved --issue`. Fresh approve correctly calls `assertParentIssueMatches` (`decomposition.ts:322`), but the idempotent short-circuit does not.

**Impact:** If a worktree retains an applied record from a prior issue session and the caller passes a different `--issue`, approve silently reports the wrong children as success. Unlikely in normal per-issue worktree usage, but inconsistent with spec language: "on read/approve, `plan.parent_issue` must equal the resolved `--issue` number."

**Recommendation:** Add `if (applied.parent_issue !== issueNumber) throw new DecompositionValidationError(...)` before the idempotent print loop. Add a CLI test.

### 🟡 Suggestion — Test gaps on approve error paths

The command test suite covers happy paths, engine gate, idempotency, validation failure, and autonomous approve, but omits:

1. **Approve parent mismatch** — preview has `parent_issue: 99`, `--issue 37` → exit `1`, no `createChildIssues`.
2. **`ChildIssueCreationError` propagation** — mock `createChildIssues` rejection → exit `1`, no applied record written.
3. **Wrong parent in child body** — preview child references `#99` for parent `37` → `ensureParentSection` throws during approve.

Task 3 unit-tests `ensureParentSection` in isolation; approve-level integration would close the loop.

### 🟡 Suggestion — `readDecompositionApplied` has no structure validation

**Location:** `src/planner/decomposition-store.ts:98-110`

Applied record is `JSON.parse`d and cast without schema checks. Corrupt or partial files throw generic parse errors or produce malformed idempotent output. Spec explicitly defers applied-record schema validation for v1, so this is acceptable but worth noting for operational robustness.

### 🟢 Nice-to-have — Minor CLI test gaps (non-blocking)

- `generate` when `runIssueDecomposer` throws `PlannerError` (handler path is straightforward via `withCommanderErrorHandling`).
- `edit` when editor exits non-zero (implemented at `decomposition.ts:277-281`, untested).
- `show` when preview JSON is valid but `parent_issue` mismatches `--issue` (spec does not require parent check on show; optional hardening).

### 🟢 Nice-to-have — `ensureParentSection` regex uses multiline `^`

**Location:** `src/github/issues.ts:36`

`PARENT_HEADING = /^## Parent\s*\n+\s*#(\d+)/m` matches anywhere in the body, not only at the start. Unlikely to cause false positives in practice; spec does not constrain position.
